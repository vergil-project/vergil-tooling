What should the output look like in CI?

When stdout is not a TTY (GitHub Actions, piped), rich's live rendering is disabled automatically. Three options for what replaces it.

▶ changelog
generating CHANGELOG.md from commits… wrote 42 entries
◀ changelog — ✓ 3.2s
▶ build-images
building ghcr.io/vergil-project/dev-base:2.1.2 Step 1/12: FROM python:3.12-slim Step 2/12: RUN apt-get install… Step 9/12: FAILED — exit 1
◀ build-images — ✗ 31.2s (deferred)
✗ release 2.1.2 completed with errors

A — GitHub Actions log groups

Emit ::group:: / ::endgroup:: annotations. Each stage becomes a collapsible section in the Actions log UI — failed stages auto-expand. Only works in GHA; plain text in other CI.

→ changelog starting…
generating CHANGELOG.md from commits… wrote 42 entries
✓ changelog 3.2s
→ build-images starting…
building ghcr.io/vergil-project/dev-base:2.1.2 Step 1/12… Step 2/12… Step 9/12: FAILED
✗ build-images 31.2s (deferred)
✗ release completed with errors — exit 1

B — Flat verbose (CI default)

Plain sequential output: start marker, all subprocess output, completion marker. No cursor tricks, no CI-specific annotations. Works identically in every CI system and when piped locally. Simple and portable.

# GITHUB_ACTIONS=true → uses groups
# piped locally → uses flat verbose
# TTY → uses rich live display
✓ TTY → rich rolling window
✓ GITHUB_ACTIONS → ::group:: annotations
✓ piped/other CI → flat verbose
--output-format [rich|gha|plain] overrides

C — Auto-detect with override

Checks $GITHUB_ACTIONS, TTY state, and a --output-format flag to pick the right renderer automatically. TTY → rich. GHA → groups. Piped/other CI → flat. One flag to override if auto-detect is wrong.

The existing output.py already detects TTY vs CI — this would extend that detection to drive renderer selection.
