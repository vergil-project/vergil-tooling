Phase 5 — two await-loops, one trivial handoff

Both agents run the same vrg-pr-await loop on the PR; only the stop conditions differ. The human's entire job is to paste one line into each session.

The one-liner handoff

On success, vrg-submit-pr prints the line to paste — e.g. /vergil:pr-watch <PR_URL>. You paste the same line into both the user-agent and audit-agent sessions. The skill reads its own identity (VRG_IDENTITY_MODE = user / audit) and runs the matching loop. As lazy as it gets — type/paste, done.

🤖 User agent loop

vergil-user[bot] · fixes
await · blocks

vrg-pr-await

Wake on any PR change (CI status, review, comment).

examine

Read PR state

CI statuses + audit review + human comments.

act

Reconcile → push

Patch all three sources → commit → push (the new commit re-enters the loop).

↺ loop on each new commit / change
STOP ✓ all checks green + audit approved → tell human "mergeable".
STOP ✗ can't get green / stuck → ask the human.

🔍 Audit agent loop

vergil-audit[bot] · reviews
await · blocks

vrg-pr-await

Wake when the user agent pushes a new commit.

review

Re-review the delta

Only the new commits · read-only · standards + suppression scrutiny.

verdict

Approve / request changes

Posts a GitHub PR review.

↺ approved-but-not-all-green ⇒ keep looping (re-review future fixes)
STOP ✓ approved AND all checks green → nothing left to wait for.
STOP ✗ error-abort: reject, "requires human" → no merge without explicit approval. Not the honor system.
The shared key: both loops re-trigger on a new commit. The user agent's push is the signal that wakes the audit agent's next review and the user agent's own next status check. The PR's commit stream is the clock.
