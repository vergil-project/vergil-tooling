The audit check

vergil-audit/approved is a check-run the App posts on the head commit — success only when it approves, failure/pending otherwise. Same act as writing approve to the local feedback file, now public.

No approval ⇒ check red ⇒ 🔒 merge blocked. GitHub enforces it, not the bot's good behavior.

Why B wins for us

  • Both bots stay GitHub Apps — no machine-user seat, identity model untouched.
  • Mirrors the local loop exactly (write-approve → post-pass).
  • Findings still posted as PR review comments = the human-readable archive.
  • Standard mechanism — it's how every CI app already gates merges.

Human override (not an agent role)

Only a human repo admin can force-merge past a red audit check — a deliberate, logged act. vergil-audit is a required check, not optional: bypass it and you're simply not using the system as designed. The reserved admin agent role stays unused.
