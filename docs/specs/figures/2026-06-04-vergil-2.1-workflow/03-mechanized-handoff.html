The handoff, mechanized — no agentic polling

Diogenes' lesson: the directory IS the state store, and deterministic code — not the AI — decides what happens next. Our .vergil/ files are already that store. So a tiny blocking-wait tool (green) replaces token-burning file-polling, and it grows into a full MCP controller later.

Human = the hard gate. Complexity-by-default is fine because you can short-circuit anytime: ignore the audit, run vrg-submit-pr on a "red", or merge. The agents only "fix each other" while you choose not to intervene. No bypass of the pipeline itself — full gates + audit run every time, even for a typo. Complementary contexts: user agent solves; audit holds it accountable.

What we borrow from Diogenes

  • Directory-as-state-store. Progress = which files exist, not what the AI claims. .vergil/ already is this.
  • A next_step oracle in Python decides whose turn it is — the AI never infers the next move from its own output.
  • Prompts stored as files, handed by reference — the agent isn't preloaded with the whole workflow.
  • Deterministic mechanization between steps (their killer feature: code that re-verifies and drops hallucinated agent output).

MVP now → controller later (same pattern, two scales)

  • MVP: two blocking-wait tools (vrg-await-audit / -template) over the .vergil/ files. Tiny, cheap, kills polling. Ships the loop today.
  • Later: a .vergil/ state machine + MCP "what next" server that hands each agent its step-prompt and advances state — the Diogenes engine, lifted.
  • Heads-up: Diogenes' own server-side execute path is still a stub (mid-surgery) — copy the pattern and the state-machine core, not that half.
