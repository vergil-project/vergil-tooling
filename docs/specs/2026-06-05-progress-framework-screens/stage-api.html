How should stages be declared in code?

This is the API every procedural command will use. Pick what feels most natural to write and read.

def run(ctx): with stage("audit", mode="fail_fast", skip_flag="skip_audit"): run_audit(ctx) with stage("build-images", mode="fail_defer"): build_docker_images(ctx) with stage("publish-pypi", mode="fail_defer"): publish_to_pypi(ctx)

A — Context manager

Each stage wraps its work in a with block. The framework catches exceptions, applies the failure mode, and handles rendering. Familiar Python idiom — same shape as with open().

STAGES = [ Stage("audit", run_audit, mode="fail_fast", skip_flag="skip_audit"), Stage("build-images", build_docker_images, mode="fail_defer"), Stage("publish-pypi", publish_to_pypi, mode="fail_defer"), ] run_pipeline(ctx, STAGES)

B — Declarative stage list

Stages declared as data, pipeline driven by a runner. Easy to inspect, reorder, or skip stages programmatically. The stage list becomes a readable spec of what the command does.

# stages defined as data (reorderable, inspectable) STAGES = [ Stage("audit", run_audit, mode="fail_fast", skip_flag="skip_audit"), Stage("build-images", build_docker_images, mode="fail_defer"), ] # runner exposes context manager for ad-hoc inline stages with pipeline.stage("preflight-check", mode="fail_fast"): assert_clean_tree(ctx)

C — Declarative list + context manager escape hatch

Main stages declared as data and run by the pipeline. Inline with pipeline.stage() available for one-off stages that don't warrant a named function. Best of both — structure where it matters, flexibility where it doesn't.

All three support the same failure modes, skip flags, and rendering. This is purely about the authoring experience.
