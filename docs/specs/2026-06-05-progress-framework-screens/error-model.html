Which stages should be allowed to fail without aborting?

Two models — the difference is whether you predeclare which failures are tolerable, or decide at runtime

✓ changelog 3.2s
✓ version-bump 1.1s
✓ tag-and-release 8.4s
⚠ build-images failed — continuing
✓ publish-pypi 4.7s
✓ docs-deploy 12.1s
⚠ release complete with 1 non-fatal failure:
✗ build-images: docker build exited 1 (dev-base:2.1.2)
full output → ~/.vrg/release-20260605.log
exit 1

A — Predeclared failure tolerance

optional stages declared in config/code
required stages abort the whole run if they fail

Each stage is statically marked optional=True or required=True. Optional failures are recorded, logged, shown in the summary, and exit 1 at the end — but don't stop subsequent stages.

$ vrg release --continue-on-error build-images,docs-deploy
✓ changelog 3.2s
✓ version-bump 1.1s
✓ tag-and-release 8.4s
⚠ build-images failed — continuing
✓ publish-pypi 4.7s
✓ docs-deploy 17.3s
⚠ 1 failure (non-fatal by flag): build-images

B — Runtime flag

--continue-on-error specifies which stages are forgiven at call time

No static declaration. Caller decides per-invocation which stages are optional. More flexible, but the CI workflow config (not the code) owns the failure model. Forgiven stages still log failures and cause exit 1.

# stage definition (code/config)
Stage("build-images", optional=True, on_failure="warn")
⚠ build-images failed (optional)
✓ publish-pypi 4.7s
⚠ 1 optional stage failed · 0 required stages failed

C — Predeclared + runtime override

default in code --force-optional to override

Stages have a default tolerance level baked in (the known-flaky ones are optional by default). A runtime flag can escalate an optional to required for stricter runs, or demote a required to optional as an escape hatch.

In all three: failures are always logged in full, always shown in the final summary, always cause a non-zero exit code.
