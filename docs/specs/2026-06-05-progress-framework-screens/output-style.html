How should a running stage's output be presented?

Three approaches — completed stages always collapse to a single summary line in all three

✓ changelog 3.2s
✓ version-bump 1.1s
⠸ build running…
building ghcr.io/vergil-project/dev-base:2.1.2 ▸ Step 4/12: RUN apt-get install -y curl git… ▸ Step 5/12: COPY requirements.txt /app/ ▸ Step 6/12: RUN pip install -r requirements.txt ▸ [full output in ~/.vrg-release-20260605.log]

A — Full stream, then collapse

All output from the running stage streams live. When the stage completes, everything collapses to a single summary line. Like watching a GitHub Actions job expand/contract.

✓ changelog 3.2s
✓ version-bump 1.1s
⠸ build
▸ Step 5/12: COPY requirements.txt… ▸ Step 6/12: RUN pip install -r requirements.txt ▸ Step 7/12: RUN pytest tests/… rolling last 3 lines · full log: ~/.vrg/release.log

B — Rolling last-N window

Only the last 3–5 lines of output are shown, updating in place. Prevents terminal flood from verbose tools (Docker, pip) while still showing live progress. Requires cursor movement.

✓ changelog 3.2s
✓ version-bump 1.1s
⠸ build 00:47 elapsed
all output → ~/.vrg/release-20260605-143022.log

C — Spinner + elapsed only

No streaming output at all. Just a spinner, the stage name, and elapsed time. All verbose output goes directly to a log file. Cleanest display — zero noise during runs.

In all three: completed stages show ✓ or ✗ with elapsed time. Non-fatal errors are flagged inline but don't stop the run. Full summary printed at the end.
